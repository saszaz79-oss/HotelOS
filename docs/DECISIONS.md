# HotelOS Architectural & Product Decisions Log

This log records major decisions made during Phase 0, including a consistency review across the Constitution, Blueprint, PRD, Architecture, Database Schema, UX System, and Roadmap. It is the reference for "why did we choose this" going forward.

## D1: Modular monolith over microservices for v0.1
**Decision**: single Next.js/Node deployable with enforced internal domain-module boundaries, not separate services.
**Reasoning**: MVP data volume and team size don't justify network-call overhead and deployment complexity of microservices; enforced module interfaces preserve a clean extraction path later.
**Alternatives considered**: microservices per domain (rejected — premature for MVP scale, per Roadmap M0–M7 timeline); single unstructured monolith with no module boundaries (rejected — violates Constitution §7 modularity standard and would make later extraction painful).
**Consequences**: extraction pipeline and AI module must still be designed as if they could become independent services (async-first, clear interfaces) so the option stays open (see Architecture §10).

## D2: Document-upload extraction, not live PMS API, for v0.1
**Decision**: HotelOS v0.1 ingests Opera Cloud data via PDF upload, not the Opera Cloud API.
**Reasoning**: API access requires commercial/technical integration agreements per hotel that would block MVP timeline; PDF export is something every Opera Cloud user already does today, lowering adoption friction.
**Alternatives considered**: Opera Cloud API integration from day one (rejected — deferred to v0.3 per Roadmap, once PDF-based value is validated).
**Consequences**: extraction accuracy is inherently bounded by PDF layout consistency; this is why the adapter architecture (Architecture §7) and mandatory manual-review step (PRD §4) exist — the product is honest about not parsing every possible layout (Blueprint Non-Goals, Constitution §1 truth test).

## D3: Deterministic adapters + generic fallback, not "AI parses everything"
**Decision**: known report types use deterministic pattern-based extraction adapters; unrecognized PDFs get a generic fallback that stores raw text but does not claim structured extraction.
**Reasoning**: an AI-only extraction approach risks silently misreading numbers in a financial context — deterministic parsing against known layouts is auditable and predictable; where it fails, the system says so rather than guessing (Constitution §2.1, §4).
**Alternatives considered**: LLM-based extraction for all report types (rejected for v0.1 as the primary path — noted as a possible future enhancement to *assist* the manual-review step, not replace deterministic parsing, if pilot data shows adapter coverage gaps).
**Consequences**: supported layouts must be explicitly documented (README, per PRD/Architecture) and the product must never imply broader support than it has.

## D4: hotelId scoping enforced in a shared data-access layer, not per-route
**Decision**: a centralized, mandatory hotel-scoping mechanism (Architecture §4) rather than trusting each route/query author to remember `WHERE hotelId = ...`.
**Reasoning**: this is the single highest-severity risk in the product (Constitution §2, non-negotiable rule #2/#3) — a hospitality SaaS leaking one client's data to a competitor is an existential failure, not a bug. Centralizing removes the failure mode of "one route handler forgot the filter."
**Alternatives considered**: Postgres row-level security (RLS) as the enforcement point (considered viable and not rejected outright — flagged as an option to evaluate in Phase 1 implementation as a *defense-in-depth addition* to, not replacement for, the application-layer enforcement, since RLS alone wouldn't cover the Super Admin bypass semantics cleanly).
**Consequences**: Phase 1 must build and test this layer before any hotel-owned feature is built on top of it; tenant-isolation tests are a quality gate, not optional (Constitution §10, PRD/Architecture §4).

## D5: Health Score is factor-transparent by construction
**Decision**: `Insight.healthScoreFactors` is a required, always-rendered part of the schema/UI — the score cannot exist in the data model without its contributing factors.
**Reasoning**: directly satisfies the explicit product requirement that the score "must not be an arbitrary decorative number" and the Constitution's "show your work" UX principle; making it structurally required (not an optional JSON field a developer might skip populating) prevents future drift toward a black-box number.
**Alternatives considered**: a simpler single computed float with methodology only in code/docs (rejected — fails the transparency requirement and the trust risk noted in Blueprint Risks).
**Consequences**: any new factor added to the score calculation must be added to the factor-breakdown output in the same change — enforced by code review discipline, not tooling, in v0.1.

## D6: AI grounding — retrieval over stored structured data, not open generation
**Decision**: the AI module only reasons over retrieved `HotelMetric`/`ReportDocument`/`Insight` records for the authorized hotel; it does not have open web access or the ability to compute new "facts" outside stored data.
**Reasoning**: directly required by Constitution §4 and PRD §8; a hallucinated occupancy number is a business-critical trust failure for a GM making staffing/pricing decisions.
**Alternatives considered**: giving the AI broader reasoning latitude (e.g., estimating unavailable metrics) — explicitly rejected; the AI must state data is missing rather than estimate.
**Consequences**: the AI's usefulness is capped by what's been uploaded/extracted — this is treated as a feature (drives report-upload behavior) not hidden as a limitation; the UI must make "why can't it answer this" self-evident (missing data → clear message, not a vague failure).

## D7: Comparisons computed on-demand, not pre-materialized, in v0.1
**Decision**: no dedicated comparison-result table; Comparison Center and Mission Control's today-vs-yesterday compute from `HotelMetric` at request time.
**Reasoning**: MVP data volume (single hotel, daily-granularity metrics) makes on-demand computation fast enough; avoids the complexity and staleness-management cost of a materialized layer before it's proven necessary.
**Alternatives considered**: pre-computed/cached comparison tables (deferred to v0.2, see Roadmap, revisited only if measured as a bottleneck).
**Consequences**: query patterns must stay indexed correctly (`HotelMetric(hotelId, metricDate)`, Database Schema §10) since correctness now depends on query-time joins being efficient.

## D8: Western Arabic numerals in both language modes
**Decision**: metric values render with 0–9 numerals regardless of UI language, per UX System §4.
**Reasoning**: consistency and scan-ability for bilingual teams comparing screenshots/exports across Arabic and English views, and to avoid ambiguity in financial figures — a deliberate, documented choice rather than a default left unexamined.
**Alternatives considered**: Eastern Arabic-Indic numerals in Arabic mode (rejected for v0.1 as the default; could become a user-level preference later if pilot feedback requests it — not precluded by the architecture).
**Consequences**: this must be applied consistently across UI, PDF export, and AI responses — flagged as a specific verification item in Quality Gates (RTL testing).

## D9: Mocked notification delivery in v0.1, real delivery in v0.2
**Decision**: password reset (and any future notification) uses a `NotificationProvider` interface with a logging/mock implementation in v0.1.
**Reasoning**: real email/SMS delivery infrastructure (deliverability, domain reputation, provider contracts) is a distraction from proving the core report→insight workflow; the interface abstraction means swapping in a real provider later is additive, not a rewrite.
**Alternatives considered**: shipping without any reset flow (rejected — Constitution/PRD require the *architecture* to support it even if delivery is mocked, so the UX and security flow — token expiry, single-use, session invalidation — are real and testable now).
**Consequences**: the mock must be clearly internal/dev-only (never presented to a real end user as "email sent" if no email will arrive) — this is a Quality Gate item ("verify no mock button is presented as functional").

## D10: RolePermission as code-level enum map, not a DB table, in v0.1
**Decision**: role→permission mapping starts as an in-code structure rather than a queryable `RolePermission` table.
**Reasoning**: the seven roles in PRD §1 are fixed and product-defined for v0.1 — no customer-facing "create a custom role" requirement exists yet. A DB table adds a migration/seed burden for a dimension that isn't actually variable yet.
**Alternatives considered**: full DB-backed RBAC from day one (deferred — revisit if/when a group customer requests custom roles; the schema note in Database Schema §1 flags this explicitly so it's a known, not forgotten, simplification).
**Consequences**: any role-permission change ships as a code change (reviewed like any other logic change), not a data change — acceptable given the fixed role list is itself a documented v0.1 scope decision.

## D11: Mission Control OS framing over SaaS dashboard framing
**Decision**: HotelOS is positioned and built as an operating system (persistent state, timeline/memory, agent-organized capability) rather than a dashboard product.
**Reasoning**: Product Owner directive, made explicit before Phase 1 to avoid the default gravitational pull toward "another admin panel" — framing decisions made after screens exist are far more expensive to unwind than framing decisions made before.
**Alternatives considered**: iterative reframing after MVP ships (rejected by Product Owner — the risk was building dashboard-shaped architecture first and retrofitting OS framing later, which the Constitution's restraint principle would flag as expensive rework).
**Consequences**: the Mission Control shell is agent-aware navigation (Architecture §13), not a fixed sidebar; `UX_SYSTEM.md` visual language will be refreshed in Phase 1 against Apple/Linear/Stripe/Tesla mission-control references before any screen is finalized (tracked as a Phase 1 task, not yet executed as of this entry).

## D12: Agent Architecture as v0.1 framework, not v0.1 feature set
**Decision**: the Agent Runtime, `Agent` interface, and Registry ship in v0.1; only the Executive Agent is a complete agent; Revenue and Front Office Agents ship as registered stubs.
**Reasoning**: building the framework against a single real agent risks designing an interface that's actually just "the Executive Agent's internals with extra steps." Registering two stub agents forces the interface to prove it generalizes before the team is deep into v0.2, at low cost (stubs, not full builds).
**Alternatives considered**: building all seven agents in v0.1 (rejected — directly violates Constitution §1 restraint test and the explicit instruction not to build all future modules at once); deferring the framework entirely to v0.2 and building the Executive Agent as a one-off in v0.1 (rejected by Product Owner — the explicit ask was enterprise-ready, expansion-ready architecture from day one, and retrofitting a shared-module-consuming framework under an already-built monolithic AI module is the exact rework the Constitution warns against).
**Consequences**: v0.1 has real engineering cost attributable to a framework with only one full consumer — accepted deliberately, tracked as the cost of the "no redesign later" requirement.

## D13: Timeline and Memory are shared domain modules, not agent-owned data
**Decision**: `timeline` and `memory` are agent-agnostic domain modules alongside `metrics`/`reports`/`insights`; agents consume them, none of them own a private copy.
**Reasoning**: if each agent maintained its own event log or memory index, cross-agent consistency (e.g., Executive Agent citing a Revenue Agent's finding later) would require reconciliation logic that the shared-module design avoids entirely; it also keeps hotelId-scoping enforcement singular (Architecture §4) rather than duplicated per agent.
**Alternatives considered**: per-agent timeline/memory namespaces (rejected — multiplies the tenant-isolation surface area that must be tested, directly working against the Constitution §10 definition-of-done requirement).
**Consequences**: timeline event taxonomy (`report.uploaded`, `alert.raised`, etc.) is a shared vocabulary all modules must publish into correctly — a schema/contract discipline enforced in code review, similar to D5's health-score-factor discipline.

## D14: Structured memory in v0.1, semantic (vector) memory deferred to v0.2
**Decision**: Hotel Memory in v0.1 is a query layer over existing structured tables (metrics history, timeline, conversations); pgvector-based semantic retrieval is v0.2+.
**Reasoning**: v0.1 pilot hotels will have weeks, not years, of history — semantic pattern retrieval over thin history offers little real value and risks the AI truthfulness rules (Constitution §4) more than a disciplined structured-filter approach does. Ships now what's genuinely useful now; the deeper capability waits for the data that makes it meaningful.
**Alternatives considered**: building semantic memory in v0.1 to "future proof" (rejected — speculative complexity ahead of validated need, the exact pattern the Constitution's restraint test exists to catch); skipping memory entirely until v0.2 (rejected by Product Owner — memory must start accumulating from v0.1 even in structured form, since a memory system is only as good as the history it has, per Blueprint).
**Consequences**: pgvector is a documented v0.2 dependency, not provisioned in v0.1 infrastructure; the `memory` module's interface is still designed in v0.1 to make adding a semantic retrieval path additive (new method, not a rewrite).

## D15: Enterprise-scale infrastructure choices provisioned in v0.1, not deferred
**Decision**: stateless app tier, connection pooling, real job queue, caching-layer seam, per-tenant rate limiting, and `(hotelId, date)` indexing discipline are all v0.1 requirements (Architecture §16), even though pilot-scale load doesn't strictly need them yet.
**Reasoning**: Product Owner directive that the architecture must "comfortably scale to 1000+ hotels without redesign." The specific items chosen are the ones that are expensive to retrofit (connection handling, indexing, statefulness assumptions baked into code) versus the ones that are cheap to add later when actually needed (multi-region, sharding — explicitly deferred, see Roadmap).
**Alternatives considered**: deferring all scale work until real load demands it (rejected — directly contradicts the explicit Product Owner requirement; also, some of these choices, like statelessness and indexing, are far cheaper to build in from the start than to retrofit after data/code exists that assumes otherwise); over-building for scale immediately (e.g., multi-region from v0.1) (rejected — Constitution §1 restraint test, and no current evidence 1000-hotel load is imminent enough to justify the operational complexity now).
**Consequences**: v0.1 infrastructure footprint (Redis, a job queue, a connection pooler) is larger than a minimal MVP would otherwise need — accepted deliberately as the cost of the no-redesign requirement, same category of tradeoff as D12.

## D16: Platform extension points (Plugin/Agent/Event) documented and partially built in v0.1, not deferred wholesale
**Decision**: the Plugin SDK manifest shape, Agent Marketplace metadata shape, and Domain Event Bus are all specified in v0.1 (Architecture §17–§19); the Event Bus is real, in-process code used internally from v0.1, while the Plugin SDK and Marketplace remain documented interfaces with no runtime.
**Reasoning**: Product Owner directive that the platform must support third-party/internal-team extension "without modifying the core" for a ten-year horizon. The Event Bus specifically earns its v0.1 place on its own merits — it already decouples `report-extraction` from `insights` internally — so building it now is good internal architecture, not speculative platform-building; the Plugin SDK and Marketplace, by contrast, have no real consumer yet (no third-party developer, no second agent in production), so building their runtime now would be exactly the premature complexity the Constitution's restraint test forbids.
**Alternatives considered**: building the full Plugin SDK loader and sandbox in v0.1 to "get ahead of it" (rejected — no plugin exists to validate the design against, and a wrong design built early is more expensive to unwind than a documented-but-unbuilt one); deferring even the Event Bus to v0.2 (rejected — it has immediate, real internal value as a decoupling mechanism between `report-extraction`, `insights`, and future genome indexing, independent of any external plugin ever existing).
**Consequences**: Architecture §26 (Platform Maturity Map) exists specifically to keep this distinction honest and current — any future PR that moves a capability from "documented interface" to "real code" must update that section in the same change (same discipline as D5's health-score-factor rule).

## D17: Agent Marketplace metadata is additive to the existing `AgentDefinition`, not a parallel structure
**Decision**: `version`, `healthStatus`, `aiProviderConfiguration`, `knowledgeSource`, `updateLifecycle` (Architecture §19) are documented as fields the v0.1 `AgentDefinition` (§13) grows into, not a separate marketplace-specific record type.
**Reasoning**: internal agents and future third-party agents must be the same mechanism (Constitution's general preference for one code path over two, applied here) — a Marketplace is not a different kind of agent, it's agents with more operational metadata attached once there's a second consumer besides the core team.
**Alternatives considered**: a separate `MarketplaceAgent` wrapper type around `AgentDefinition` (rejected — creates exactly the two-code-paths problem the Agent Runtime was built in D12 to avoid).
**Consequences**: when Revenue Agent goes from stub to full implementation (Roadmap v0.2), extending its registry entry with real `healthStatus`/`aiProviderConfiguration` is additive to the existing struct, not a migration to a new one.

## D18: Hotel Genome renames and expands Hotel Memory; no code migration needed
**Decision**: "Hotel Genome" replaces "Hotel Memory" as the product and architecture term, with expanded scope (seasonal/guest/revenue/occupancy behavior, operational strengths/weaknesses, recurring issues, employee performance trends, historical decisions, AI recommendation outcomes) per Product Owner direction.
**Reasoning**: at the time of this rename, the `memory` domain module existed only as a documented interface (Architecture §15 prior revision) with no implemented code — Report/Metric/Timeline/AIConversation tables existed, but no `memory` module code had been written yet. This makes the rename free: update documentation and the (not-yet-written) module name, no data migration, no breaking change to running code.
**Alternatives considered**: keeping "Memory" as the technical/schema term and "Genome" as marketing-only language (rejected — Constitution's "no hidden prompts/no dishonest naming" spirit extends to internal consistency; a codebase that says `memory` while every doc and the product itself says "Genome" is exactly the kind of drift that makes documentation untrustworthy over time).
**Consequences**: any future PR implementing the `genome` module uses that name from its first commit; the expanded scope (recommendation-outcome tracking, behavioral pattern modeling) is explicitly staged across v0.1 (structured substrate + outcome field) → v0.2 (semantic retrieval) → v0.3 (behavioral modeling proper) rather than promised all at once (Architecture §15).

## D19: Prompt Registry starts file-based and version-controlled, not database-backed
**Decision**: v0.1 prompts live as versioned files under `src/server/modules/ai-orchestration/prompts/` with a metadata header, not in a `Prompt` database table with an approval-workflow UI.
**Reasoning**: satisfies the actual requirement — "never hidden, always version-controlled" — using the version control system that already provides exactly that (author, timestamp, diff, history) for free. A database-backed registry with an approval UI is real value once there are enough prompts across multiple agents that non-engineers need to review/approve them without reading a git diff; with one agent and a handful of prompts in v0.1, that UI would be unused scaffolding.
**Alternatives considered**: database-backed registry from v0.1 (rejected — no current user of an approval UI; git history already satisfies "change history" and "never hidden" for the current scale); no versioning discipline at all, prompts as inline strings (explicitly rejected by Product Owner directive).
**Consequences**: the metadata shape (owner, version, language, modelCompatibility, variables, approvalStatus) is still defined now as a header convention, so migrating to a DB-backed registry later (Roadmap v0.2) is a storage change, not a schema redesign or a new set of fields to invent under time pressure.

## D20: Knowledge Registry schema documented, deliberately unpopulated in v0.1
**Decision**: `KnowledgeSource` shape is specified (Architecture §22) but no rows exist and no agent references it in v0.1 — the Executive Agent grounds only in hotel-specific data.
**Reasoning**: v0.1 has exactly one agent, and it doesn't need generic domain-knowledge grounding (its job is hotel-specific executive analysis, not SOP compliance guidance) — building a knowledge ingestion/management system for zero current consumers is speculative. The shape is documented now so that when Front Office or Housekeeping Agents (which plausibly do need SOP grounding) arrive in v0.2/v0.3, they have a pattern to plug into rather than inventing one under deadline pressure.
**Alternatives considered**: seeding placeholder knowledge sources in v0.1 "for completeness" (rejected — Constitution §2 rule #16 requires seed/demo data to be clearly labeled and purposeful; placeholder knowledge content serving no real agent would be exactly the kind of decorative scaffolding the restraint test exists to prevent).
**Consequences**: no additional Phase 1 implementation work from this decision — it is documentation-only until a knowledge-grounded agent is actually built.

## D21: Workflow Automation Engine is architecture-preparation only, per explicit Product Owner scope boundary
**Decision**: no workflow data model, execution runtime, or builder UI is built in v0.1 or planned before v0.4 (Roadmap). Only the Domain Event Bus (D16) and the provider-abstracted `NotificationProvider` interface (D9) — both independently justified — serve as the substrate a future engine would consume.
**Reasoning**: directly stated by the Product Owner ("This does NOT need implementation now. Only prepare the architecture.") — the clearest-scoped decision in this revision, requiring no alternatives analysis.
**Consequences**: Architecture §20 is written so a future engine's design starts from "wire a trigger UI to already-existing typed events and already-existing notification/export actions," not from "first invent an event system" — the preparation is real even though the engine is not.

## D22: v0.1 implementation scope narrows to one workflow; agent/adapter breadth explicitly deferred
**Decision**: per Product Owner direction ("Quality over quantity. Build one complete workflow first."), v0.1 implementation targets exactly one deterministic extraction adapter (Manager Flash) rather than all four originally planned, and no further work proceeds on Revenue/Front Office Agent beyond their existing registry stub until the core workflow (upload → validation → extraction → normalization → Mission Control → AI analysis → PDF export → archive) is production-quality.
**Reasoning**: a further tightening of the adapter-scope reasoning already established in D2/D3 — building three more adapters or two more agents in parallel with the first one reaching real quality would repeat the "breadth over depth" mistake the Constitution's restraint test exists to prevent, now made explicit by direct Product Owner instruction rather than inferred from general principle.
**Alternatives considered**: keeping all four adapters and the stub-agent validation work in v0.1 as originally scoped in the M0 revision (superseded — explicitly overridden by this directive); removing the already-built Revenue/Front Office stub registrations entirely to minimize v0.1 surface area (rejected — they cost nothing further to leave in place as inert registry entries and already demonstrate the Agent Runtime generalizes per D12's original goal; removing working code to satisfy a scope directive that only asked to stop *adding* work would be needless churn).
**Consequences**: `MVP_PRD.md` and `ROADMAP.md` both updated to reflect Manager Flash as the sole v0.1 adapter target, with Reservation Statistics/Open Balance/Reservation Statistics 1 moved to v0.2.

## D23: DDD bounded contexts are a documentation mapping over existing modules, not a file-tree rename
**Decision**: Architecture §27 names bounded contexts (Identity, Hotels, Reports, Metrics, Executive Intelligence, Revenue, Housekeeping, Finance, AI Platform, Automation, Notifications, Audit) and maps each to current/future modules, without renaming or restructuring the `src/server/modules/*` folder layout.
**Reasoning**: the module-interface discipline already required since §2 (explicit exported functions, no cross-module table access) is what actually delivers DDD's value — a bounded context with a protected internal model. A folder rename to `src/server/contexts/<context>/<module>` would cost real effort and touch every existing file for zero behavioral change, which is precisely the kind of premature refactor the Constitution's restraint test exists to catch, especially since several named contexts (Housekeeping, Finance) have no code at all yet to move.
**Alternatives considered**: physically reorganizing the codebase now while it's small (rejected — "cheap because it's small" is true today and will be equally true whenever a context actually needs its own repo/service; doing it speculatively now buys nothing a later, purposeful reorg wouldn't); not naming bounded contexts at all and only relying on module boundaries (rejected — Product Owner explicitly requested DDD vocabulary, and naming the contexts now gives Revenue/Housekeeping/Finance a place to be designed into later without inventing the taxonomy under deadline pressure then).
**Consequences**: new modules from M3 onward are still built inside their named context's conceptual boundary even though the folder path doesn't say so explicitly; a future physical reorg (if ever justified) is a mechanical mapping exercise against this table, not a redesign.

## D24: CQRS readiness is a naming/file convention, not infrastructure
**Decision**: commands and queries are exported as distinct, non-overlapping functions per module (Architecture §28), organized into separate `commands.ts`/`queries.ts` files once a module has enough of each to warrant it; no event sourcing, no read replica, no denormalized projection is built.
**Reasoning**: directly matches the Product Owner's own framing — "prepare... do not over-engineer the MVP." The real value of CQRS at this stage is architectural discipline (never let a query mutate, never let a command return a bloated read payload), which costs nothing and is exactly what makes a later real split cheap; the infrastructure half of CQRS is only justified by a read/write scaling mismatch this platform doesn't have at pilot scale.
**Alternatives considered**: building a real CQRS read model (separate query-optimized tables/cache) now (rejected — no evidence of the read/write imbalance that would justify it; Constitution restraint test); ignoring CQRS entirely until it's "needed" (rejected — the naming convention is free and retrofitting consistent command/query naming across a grown codebase later is genuinely more expensive than establishing it now, unlike the infrastructure half).
**Consequences**: `src/server/modules/reports` adopts the `commands.ts`/`queries.ts` split starting with the M3 work in this session, as the first applied instance; earlier small modules keep single files since the convention is about file organization scaling with module size, not a mandate to split trivially small modules.

## D25: Feature Flags reuse `HotelSetting`, no new table
**Decision**: `isModuleEnabled(hotelId, moduleKey)` (Architecture §29) is backed by the existing `HotelSetting` key-value table, not a new `FeatureFlag` model.
**Reasoning**: `HotelSetting` was designed in Phase 0 specifically as flexible per-hotel configuration without schema churn per new setting (Database Schema §8) — a module-enablement list is exactly that kind of setting. Building a dedicated table would duplicate a mechanism that already exists and already satisfies the requirement.
**Alternatives considered**: a dedicated `FeatureFlag` table with richer metadata (rollout percentage, per-role flags) (rejected for v0.1 — no current requirement needs anything beyond a per-hotel on/off list; the richer version is a natural v0.2 evolution if enterprise customers need finer-grained control, at which point `HotelSetting`'s JSON value can grow in place or graduate to a dedicated table without breaking `isModuleEnabled`'s call signature).
**Consequences**: unlike the Plugin SDK/Marketplace/Workflow Engine (documented-only, D16), Feature Flags is real, working code in this revision, wired into the Agent Runtime and the Reports upload path as concrete enforcement points — not just a documented interface, because the Product Owner's phrasing ("Introduce Feature Flags") was an implementation directive, not a preparation one, and the cost of doing so is genuinely near-zero given the existing table.

## D26: AI Orchestration Layer responsibilities formalized now; implementation stays M6
**Decision**: Architecture §30 enumerates the orchestration pipeline (retrieval, prompt selection, tool calling, reasoning, validation, citations, formatting, future multi-model routing) explicitly, superseding the looser description in the original §8, without building any of it yet.
**Reasoning**: the rule "AI never calls business services directly" was already implicit in the original module design (§8's "provider-agnostic assistant" framing, and §13's Agent Runtime pattern), but implicit rules get violated by whoever writes the AI code fastest under M6 deadline pressure if they're not written down as a checklist first. Writing the pipeline stages down now costs a documentation pass; writing them down after the first (probably rule-violating) AI integration attempt costs a rewrite.
**Alternatives considered**: waiting until M6 to design the orchestration pipeline in detail (rejected — the whole point of a Product Owner directive landing now is to prevent exactly that "design it under pressure" failure mode).
**Consequences**: M6 implementation is scoped against this checklist directly; Architecture §26's maturity map still correctly lists `ai-orchestration` as tier-2 (documented interface) until M6 ships it.

## D27: Decision Engine is the existing entity chain, reframed — not a new subsystem
**Decision**: Architecture §31 states the Data→Metrics→Insights→Recommendations→Priorities→Decisions→Actions pipeline maps onto `ReportDocument`→`HotelMetric`→`Insight`→`Recommendation`(+`.priority`)→`TimelineEvent(decision_logged)`→future `Action`, rather than introducing a new "Decision Engine" table or service.
**Reasoning**: every stage except Actions (D28) already existed in the Phase-0 schema before this directive — `decision_logged` was reserved as a `TimelineEventType` from the original OS/Agent pivot specifically because a Hotel Timeline is, definitionally, a record of things including decisions. Naming the pipeline explicitly is valuable (it's a clearer product narrative and a checklist for "is this recommendation traceable"), but building a parallel subsystem to track what the existing tables already track would create two sources of truth for the same fact.
**Alternatives considered**: a dedicated `Decision` table separate from `TimelineEvent` (rejected — `decision_logged` timeline events with a `sourceRef` to the `Recommendation` already capture this without duplicating the timeline's existing provenance/audit-adjacent role).
**Consequences**: no schema change from this decision; M5/M6 implementation of Recommendations must ensure the `insightId`/`sourceReportDocumentId` chain is actually populated correctly (it was already planned to be), since D27 relies on that chain being real, not aspirational.

## D28: Action Center stays fully unimplemented, extending D20's reasoning
**Decision**: no schema, table, or code for Actions in this revision — only the documented shape in Architecture §32.
**Reasoning**: identical reasoning to D20 (Knowledge Registry) — Actions attach to Recommendations, and Recommendations don't exist until M5. Adding an `Action` table today would be a foreign key to a table whose own producer isn't built yet, the clearest possible case of the "decorative scaffolding" the Constitution's restraint test forbids.
**Alternatives considered**: adding the table now as inert schema (rejected — same reasoning as D20; an empty, unreferenced table is worse documentation than a markdown section, since it implies false readiness to anyone reading the schema without the accompanying doc).
**Consequences**: Action Center schema work is explicitly sequenced to land alongside or after M5 (Recommendations), tracked in Roadmap Future Integrations, not silently forgotten.

## D29: Data Quality Engine is real code in M3, not deferred
**Decision**: `completenessScore`, `validationStatus`, and `qualityNotes` are added to `ReportDocument` in this revision (`extractionConfidence` already existed in the Phase-0 schema and is now given a real populating methodology instead of remaining an unused placeholder float).
**Reasoning**: unlike the other nine principles in this directive, Data Quality scoring is not preparation for a future capability — it is required by the validation stage of the workflow already on the Roadmap for this exact milestone (M3), and explicitly required as an AI grounding input for M6. Deferring it would mean building the validation stage twice: once now without scores, once later to retrofit them.
**Alternatives considered**: shipping M3's validation stage with only pass/fail plausibility checks and adding quality scoring later (rejected — Product Owner directive is explicit and immediate: "AI must use these scores when generating recommendations," which requires the scores to exist before M6, i.e., from M3 onward).
**Consequences**: this is the one schema change in this revision's implementation phase; all others (DDD, CQRS, Feature Flags reusing `HotelSetting`, AI Orchestration, Decision Engine, Action Center, Digital Twin) are documentation-only or reuse existing tables.

## D30: Digital Twin is a pure documentation placeholder — zero artifacts
**Decision**: Architecture §34 contains one paragraph and reserves nothing structurally — no table, no module stub, no naming convention.
**Reasoning**: the most explicit "do not implement" instruction in this directive, and correctly so — a Digital Twin's actual requirements (what scenarios, what inputs, what simulation approach) are unknowable until the Hotel Genome has enough real accumulated history to simulate against meaningfully (itself a v0.2/v0.3-plus concern per Architecture §15's own staging). Reserving a stub module or table now would be committing to a shape with no information to base it on.
**Alternatives considered**: none seriously — this is the clearest-scoped item in the directive.
**Consequences**: no Roadmap milestone references Digital Twin implementation; it appears only in Future Integrations (unscheduled) and this decisions log, which is the correct level of commitment for an idea with zero current design basis.

## D31: Performance Budget sets targets now; automated measurement is deferred
**Decision**: Architecture §35 publishes concrete numeric targets (cold start, dashboard load, AI response, PDF generation, upload processing, memory, bundle size) without building continuous performance monitoring/regression-testing infrastructure to enforce them automatically.
**Reasoning**: targets are cheap to state and valuable immediately — they're a design constraint every future PR should be written against (e.g., M3's extraction pipeline target of <15s end-to-end shapes whether synchronous or job-queued processing is acceptable at a given data volume). Automated enforcement (APM, regression CI gates) requires real usage data to calibrate thresholds that don't either get ignored (too loose) or false-alarm constantly (too tight, guessed from nothing) — exactly the "prepared, not wired" distinction already established for tracing/metrics in Architecture §23.
**Alternatives considered**: no targets until there's infrastructure to measure them (rejected — a target that shapes design decisions today is valuable even before it's automatically enforced; Constitution §8 already had a dashboard-load target in Phase 0, this extends the same practice); building full APM now (rejected — no current usage data to calibrate against, real infrastructure cost for a pilot-scale product, same reasoning as §23's tracing deferral).
**Consequences**: the bundle-size target is checked manually via `npm run build` output today (already demonstrated in this session's verification steps); other targets have no current measurement at all and are aspirational design constraints until Roadmap v0.2's load-validation pass (Architecture §16, Roadmap v0.2).

## D32: Product Principle elevated as the single highest-authority product gate, subsuming the existing Decision Test
**Decision**: Constitution §1 now states the Product Principle ("Does this help hotel management make better decisions?") as an explicit named gate above the three existing tests, with the Decision Test cross-referenced as its concrete application.
**Reasoning**: the Decision Test already asked essentially this question ("does this help someone decide or act faster/better?") — the Product Owner's phrasing is materially the same idea restated with more authority ("If not, it should not become part of HotelOS"). Rather than creating a fourth, redundant test, the existing test is explicitly framed as the applied form of the newly-stated principle, keeping the Constitution internally consistent instead of accumulating near-duplicate rules over successive revisions.
**Alternatives considered**: adding a fourth independent test alongside the existing three (rejected — Decision Test and Product Principle are the same question at two altitudes; treating them as separate tests would invite them drifting apart over time, which is worse than acknowledging the overlap explicitly now).
**Consequences**: any future feature-scope discussion can cite "the Product Principle" as the highest-level framing and "the Decision Test" as its concrete per-feature checklist item — both point to the same governing question, by design.

## D33: Validation Workspace is cross-hotel, Super-Admin-only, and English-only
**Decision**: `/validation` (Validation Phase §1) lives outside the hotel-scoped `(app)` shell, queries across all hotels rather than the caller's own hotel, is gated to `isSuperAdmin` only, and is not localized to Arabic.
**Reasoning**: this is explicitly framed by the Product Owner as "an internal engineering tool (not for normal hotel users)" — the three departures from every other screen in the product (cross-hotel querying, Super-Admin-only gating, no bilingual investment) all follow directly from that framing. Cross-hotel because an engineer judging parser quality needs to see patterns across the install base, not one hotel at a time; Super-Admin-only because no hotel-level role should see other hotels' data (Constitution §2 rule #12); English-only because the Constitution's bilingual requirement (§8) exists to serve the hotel-facing Arabic-first audience, who never see this surface.
**Alternatives considered**: making it hotel-scoped like everything else (rejected — defeats the purpose of judging parser quality across multiple hotels' real data); bilingual (rejected as a use of effort with no real audience — can be added later if a non-English-reading engineer ever needs it, at negligible cost given the dictionary pattern already established elsewhere).
**Consequences**: `src/server/modules/validation/queries.ts` is explicitly documented as the one query module in the codebase that does NOT enforce hotel-scoping itself — callers (currently only the `/validation` route tree) are solely responsible for the Super-Admin gate, and this must never be relaxed to a general-purpose query module reused elsewhere without re-adding tenant scoping.

## D34: "Verified" status is earned only by human confirmation, never by extraction confidence alone
**Decision**: `ExtractedField.status` can only become `'verified'` when a user manually confirms/corrects a value (`updateExtractedField`); automatic extraction, however high its internal confidence score, always lands as `needs_review` (or `ambiguous`/`missing`).
**Reasoning**: directly implements the Constitution's new "Never Hide Uncertainty" rule. The adapter's own confidence score (0.6 baseline) is a statement about pattern-match certainty, not about correctness against the real report — conflating the two would let a technically-confident-but-wrong extraction present itself as "verified," which is exactly the failure mode the Validation Phase exists to prevent.
**Alternatives considered**: a confidence threshold above which a field auto-promotes to "verified" (e.g., confidence > 0.8) (rejected — no accuracy data yet justifies picking any threshold as meaningfully trustworthy; per `VALIDATION_REPORT.md`, accuracy is unmeasured, so any such threshold would itself be a fabricated claim of reliability).
**Consequences**: every Manager Flash extraction, even a clean unambiguous match, requires a human review step before HotelOS will call it "verified" — a deliberate friction that will be revisited once real accuracy data (Roadmap Validation Phase, `VALIDATION_REPORT.md` §7) justifies loosening it.

## D35: Real Data Mode ground truth is per-report, engineer-entered, not sourced from a bulk import
**Decision**: `ExtractionGroundTruth` is one row per `ReportDocument`, entered manually through the Validation Workspace form — no bulk-import mechanism, no automatic ground-truth derivation.
**Reasoning**: ground truth, by definition, must come from a trusted source independent of the system being validated — the only trustworthy source available is a human reading the original PDF. Building a bulk-import format before any real sample exists to design the format against would be speculative (Constitution restraint test) — the form-per-report is the minimum viable mechanism to unblock `VALIDATION_REPORT.md` §7's first recommended step.
**Alternatives considered**: none seriously considered — bulk tooling is reasonably deferred until manual entry proves too slow at real validation volume, which is not yet known.
**Consequences**: validating N reports requires N manual data-entry sessions; acceptable for the "3-5 real samples" scale `VALIDATION_REPORT.md` recommends as the immediate next step, revisit if real validation volume grows.

## D36: Platform Owner is `User.isSuperAdmin`, not a fourth HotelRole
**Decision**: "Platform Owner" (Product Owner's term) maps directly onto the existing `User.isSuperAdmin` flag and the Super Admin path (Architecture §4) established since Phase 0 — no new role enum value, no new table.
**Reasoning**: the three-tier model the Product Owner described (Platform Owner / Hotel Admin / Hotel Users) is exactly the model already built: `isSuperAdmin` (cross-hotel, no membership rows) / `HOTEL_ADMIN` role / the other five `HotelRole` values. Renaming or restructuring this to introduce new terminology would be pure churn — the vocabulary changed, the architecture didn't need to.
**Alternatives considered**: a dedicated `PlatformOwner` table separate from `User` (rejected — a Platform Owner is still a user who logs in with username/password like everyone else; a separate table would duplicate auth machinery for no benefit).
**Consequences**: `/admin` is gated on `user.isSuperAdmin`, identical to the Validation Workspace's gate — same mechanism, two different consumers, consistent with D33's precedent that Super Admin gating is one reusable pattern, not reinvented per screen.

## D37: `/admin` is bilingual; `/validation` stays English-only — the distinction is who the audience is
**Decision**: the Super Admin Console uses the same `getDictionary`/RTL pattern as the hotel-facing product; the Validation Workspace (D33) remains English-only.
**Reasoning**: `/validation` is explicitly framed as an internal engineering tool no real customer ever sees. `/admin`, by contrast, is the Platform Owner's own daily-use console — and the Platform Owner in this product's actual target market (Saudi Arabia/GCC) is plausibly an Arabic-first user. Constitution §8's bilingual requirement exists to serve real users of the product; the Platform Owner is a real user of `/admin` in a way no one is a real "user" of `/validation`.
**Alternatives considered**: English-only admin console for faster delivery (rejected — directly contradicts the Arabic-first commitment for a console the Product Owner themselves will use daily; the incremental cost of using the existing dictionary pattern is low since the pattern already exists).
**Consequences**: every `/admin` string added in this revision has both `en` and `ar` entries; any future admin section must follow the same discipline, unlike anything added under `/validation`.

## D38: Admin commands enforce `isSuperAdmin` inside the command, not just at the route/layout
**Decision**: `hotels/commands.ts` and `users/commands.ts` each take an `actingUser: User` parameter and throw `FORBIDDEN` internally if `!actingUser.isSuperAdmin`, rather than trusting the calling route/layout to have already checked.
**Reasoning**: every other sensitive boundary in this codebase (hotel-scoping via `assertHotelAccess`, agent access via `requireAgentAccess`) is enforced centrally in the command/query layer, not only at the UI layer — Architecture §4's core discipline is "the enforcement point is the data-access layer, not a per-route habit." Platform administration commands are exactly as sensitive as hotel-scoping (arguably more so — they can create hotels and reset any user's password) and deserve the identical discipline, not a weaker one just because `/admin`'s layout already checks.
**Alternatives considered**: relying solely on the `/admin` layout's redirect-if-not-super-admin check (rejected — that's the Validation Workspace's model per D33's own caveat that its query module is "the one query module that does NOT enforce hotel-scoping itself"; admin commands are higher-stakes than validation queries and shouldn't inherit that weaker pattern).
**Consequences**: even a future bug that let a non-Super-Admin route call `createHotelWithAdmin` directly (e.g., a mis-wired API route) would still be blocked at the command itself — defense in depth, not defense in one place.

## D39: Admin-created and reset passwords are always random and temporary, never admin-chosen
**Decision**: `createHotelWithAdmin`, `createUser`, and `resetUserPassword` all generate a random temporary password via `generateTemporaryPassword()` (96 bits of entropy) and set `mustChangePassword: true` — there is no code path where a Super Admin types in a password for someone else.
**Reasoning**: directly implements "first login must require changing the temporary password" and "never store plaintext passwords" together — if an admin could choose a user's password, that admin would necessarily know it, which is a weaker security posture than a random, immediately-superseded temporary credential the admin merely relays once. It also removes the temptation to reuse a weak, memorable default across many accounts.
**Alternatives considered**: letting the admin set an initial password directly (rejected — weaker security model, and inconsistent with the temporary-password/forced-change flow the Product Owner explicitly required).
**Consequences**: the one-time password is displayed exactly once in the admin UI (`state.result`, not persisted client-side or logged anywhere in plaintext) — the admin's responsibility is to relay it securely immediately, since it cannot be retrieved again (only reset again, which invalidates it and issues a new one).

## D40: Migration discipline formalized now, baseline migration not yet generated
**Decision**: Architecture §9 "Migration Discipline" states the required workflow (`prisma migrate dev` for all changes, `prisma migrate deploy` in production, additive-first design) — but no `prisma/migrations/` directory exists yet in this repository, because no live PostgreSQL database has been available in this development environment at any point in this project's history to diff against.
**Reasoning**: honesty over appearance of completeness (Constitution "Never Hide Uncertainty") — claiming a migration history exists when it doesn't would be exactly the kind of fabrication the Validation Phase (previous session) was built to prevent in the extraction pipeline; the same standard applies to infrastructure claims.
**Alternatives considered**: hand-writing a migration SQL file without a real database to validate it against (rejected — an unvalidated hand-written migration is worse than an honestly-absent one; it could contain errors that only surface when someone actually tries to apply it).
**Consequences**: the first real deployment must run `prisma migrate dev --name init` once against a live database to establish the baseline (documented in Architecture §9 and README) — this is a known, tracked setup step, not a silent gap.

## D41: Next.js 14→15 and Prisma 5→7 upgraded together, forced by Cloudflare compatibility
**Decision**: upgraded Next.js to 15.5.20 and Prisma to 7.8.0 as a prerequisite for Cloudflare Workers deployment, not as independent modernization.
**Reasoning**: `@opennextjs/cloudflare`'s peer dependency requires Next.js ≥15; Prisma 7 is the first version whose driver-adapter path works without a Rust query-engine binary (Workers can't execute native binaries). Neither was optional once "deploy to Cloudflare Workers as a full-stack app" was the explicit requirement — verified empirically, not assumed (installing the adapter against Next 14 failed with a peer-dependency error; generating Prisma 5's schema against a Prisma 7 CLI is a different, non-issue — the schema itself needed the `datasource.url` removal Prisma 7 requires).
**Alternatives considered**: pinning an older, Next-14-compatible OpenNext version (rejected — the user explicitly asked for current official packages, and an old pinned adapter version is exactly the kind of hidden technical debt the Validation Phase's "Never Hide Uncertainty" principle argues against); Prisma Accelerate instead of driver adapters + Hyperdrive (rejected — Accelerate is a paid Prisma-hosted proxy, an extra vendor/cost the user didn't ask for, when Cloudflare's own Hyperdrive already solves the same pooling problem for free at pilot scale).
**Consequences**: the official `@next/codemod next-async-request-api` handled the mechanical `cookies()`/`params` migration across 30 files; manually verified the auth session module's rewrite. Full production build and the actual `opennextjs-cloudflare build` both succeeded after the upgrade.

## D42: unpdf replaces pdf-parse
**Decision**: swapped PDF text extraction libraries entirely rather than attempting to force `pdf-parse` to work on Workers.
**Reasoning**: direct inspection of the installed `pdf-parse` package's compiled output (both its Node and its own "browser" build) showed `node:worker_threads` references — unsupported on Workers even with `nodejs_compat`. `unpdf` is built and documented specifically for edge/Workers environments (worker inlined into the bundle, no `worker_threads`), verified by inspection and by successfully extracting real Arabic+English text from an actual local PDF.
**Alternatives considered**: patching/forking `pdf-parse` (rejected — maintaining a fork of a PDF-parsing library is exactly the kind of scope creep the Constitution's restraint test forbids, when a purpose-built alternative exists and is a drop-in replacement given the adapter interface already abstracted PDF extraction behind `PdfPage[]`/`RawExtractionResult`); giving up per-page text tracing (rejected — `unpdf`'s `extractText(pdf, { mergePages: false })` returns exactly the per-page array the Validation Phase's source-page tracing feature needs, so no functionality was lost).
**Consequences**: `docs/PARSER_DOCUMENTATION.md` updated to reflect the new extraction library's behavior; the adapter interface itself (`ExtractionAdapter`) required zero changes, confirming the original abstraction boundary was drawn correctly back in Roadmap M3.

## D43: R2 native binding over S3-compatible SDK
**Decision**: `src/server/modules/storage/r2.ts` uses R2's native Worker binding (`env.HOTELOS_BUCKET.put/get/delete`), not the AWS SDK against R2's S3-compatible REST endpoint.
**Reasoning**: the code already runs inside a Cloudflare Worker (via OpenNext) — the native binding is authenticated by the Worker's own identity with zero credentials to manage, rotate, or leak, versus the S3-compatible path which would require generating and securing R2 API access keys for no benefit given the calling code is already inside Cloudflare's own infrastructure.
**Alternatives considered**: `@aws-sdk/client-s3` against R2's S3-compatible endpoint (rejected — only actually necessary for code running *outside* Cloudflare, e.g. a script on the user's own machine; not HotelOS's situation, and it would add a real dependency plus request-signing complexity for no gain).
**Consequences**: no AWS SDK dependency in the project at all; `STORAGE_DRIVER=r2` requires only a binding name (`STORAGE_R2_BINDING`), not access keys — reflected in `.env.example` and `docs/R2_SETUP.md`.

## D44: Cloudflare Workers Builds recommended over GitHub Actions as the primary CI path
**Decision**: `docs/CLOUDFLARE_DEPLOYMENT.md` presents Workers Builds (dashboard-connected-to-GitHub, no YAML) as the default path, with a disabled-by-default GitHub Actions workflow (`.github/workflows/deploy.yml`) as an explicit, documented alternative.
**Reasoning**: directly matches the user's own stated preference order (Workers Builds first, GitHub Actions second) and their stated difficulty configuring GitHub — Workers Builds needs zero YAML authoring or secret wiring for the deploy step itself (only for the separate migration/seed steps, which inherently need to run outside the Worker regardless of which CI drives the deploy).
**Alternatives considered**: GitHub Actions as the only documented path (rejected — more setup steps, more secrets to manage, directly against the user's stated preference and skill comfort).
**Consequences**: migrations and production seeding still use GitHub Actions (`migrate.yml`, `seed-production.yml`) regardless of which deploy path is chosen, since Workers Builds has no equivalent "run this arbitrary one-off command against production" primitive — this is the one place GitHub Actions is not optional, only the full-deploy workflow is.

## D45: Compatibility findings verified by direct inspection/testing wherever possible, not asserted from documentation alone
**Decision**: every incompatibility claim in `docs/CLOUDFLARE_COMPATIBILITY_REPORT.md` is backed by something actually run in this environment (`grep` on compiled output, a real `opennextjs-cloudflare build`, a real PDF text extraction, a real Proxy-forwarding test) — documentation research (via WebFetch against Prisma's and Cloudflare's own current docs) was used only to determine the *correct pattern to implement*, never as the sole basis for claiming something works.
**Reasoning**: directly extends the Validation Phase's "Never Hide Uncertainty" principle to this session's own claims. A compatibility report that says "should work per the docs" without attempting to verify it would itself be exactly the kind of unearned confidence the Constitution forbids — and doing so paid off directly: the CHANGELOG.md filesystem-read bug and the `initOpenNextCloudflareForDev()`-breaks-builds issue were both found only because an actual build was run and its actual output inspected, not because either was anticipated from reading documentation.
**Consequences**: the report's "What Has Not Been Verified" section is exactly as long as it needs to be — no live database query, no live R2 write, no live Workers request has actually happened, and the report says so plainly rather than implying otherwise.

## D46: Recommendation.owner/timeframe/expectedOutcome are new nullable columns, never backfilled
**Decision**: the Analytics fix's Phase 6 migration (`prisma/migrations/20260721173201_recommendation_owner_timeframe_outcome`) adds `owner HotelRole?`, `timeframe String?`, `expectedOutcomeEn String?`, `expectedOutcomeAr String?` to `Recommendation`, all nullable, with no backfill of existing rows. `owner` reuses the existing `HotelRole` enum rather than a new free-text/enum type, replacing the display-time-only `inferSuggestedOwner()` heuristic in `insights/morning-brief.ts` (which read `Recommendation.supportingMetrics` on every render because no persisted field existed) — the equivalent logic now runs once, deterministically, in `insights/rules.ts` at rule-evaluation time, and is written directly to the new column alongside `textEn`/`textAr`.
**Reasoning**: the Constitution's "never fabricate, only flag" rule applies to schema migrations exactly as it applies to extracted report data — recommendations created before this migration ran have no real owner/timeframe/expected-outcome on record, so guessing one at read time (even via the same heuristic morning-brief.ts used to run) would be indistinguishable from a fabricated value once persisted-looking UI (the Executive Export Action Plan table) renders it next to genuinely-computed KPIs. Legacy rows show "not yet available" (`dict.executiveExport.notYetAvailable`) instead.
**Alternatives considered**: backfilling old rows by re-running the current rule logic against their stored `supportingMetrics` (rejected — the rule logic has since changed shape multiple times across Phases 3-6, so a "backfilled" value wouldn't actually reflect what the rule would have produced at the time, which is the same trap as fabricating it fresh); a free-text `owner` column instead of reusing `HotelRole` (rejected — `owner` only ever takes one of three `HotelRole` values in practice, `REVENUE_MANAGER`/`FRONT_OFFICE_MANAGER`/`GENERAL_MANAGER`, and reusing the enum keeps it in sync with the one HotelOS already uses for membership roles instead of maintaining a second, driftable vocabulary).
**Consequences**: `insights/rules.ts`'s `RuleRecommendation` interface now requires `owner`/`timeframe`/`expectedOutcomeEn`/`expectedOutcomeAr` on every rule family (mechanically enforced by TypeScript, matching the existing "no generic phrases" numeric-interpolation discipline); a new `dict.hotelRoles.*` dictionary section was added (both locales) since `HotelRole` values were previously shown raw in `mission-control/page.tsx`'s Morning Brief owner line — now labeled there too, not just in the new Action Plan table; the migration itself is committed but not applied to production until run through the existing manually-gated `migrate.yml` workflow (typed "migrate" confirmation), per the standing migration discipline (D40).
